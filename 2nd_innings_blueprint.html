Section 03
Tech stack — lean enough for one Flutter dev
Every tool chosen here has a specific job and integrates cleanly with Flutter. No AWS, no Kubernetes, no custom backend. A single full-stack Flutter developer with Supabase familiarity can own this entire stack end to end.
Flutter
App framework · iOS + Android
Single codebase for both platforms. Mature UAE freelance market. Deploy to App Store and Play Store from one project. Chosen over React Native because Flutter's widget rendering is more consistent for senior-friendly UI where large text and tap targets matter.
Supabase
Backend · Auth · DB · Storage
Postgres + Auth + Storage + Realtime in one. Official Flutter SDK. Handles phone OTP, user data, photo storage, and admin table editing — no custom admin UI needed. One dev can manage the full backend.
Agora Chat SDK
Group chat · neighbourhood groups
Purpose-built chat SDK with Flutter support. Handles group creation, message delivery, media sharing, read receipts, typing indicators. Offloads all real-time chat complexity from Supabase. Scales to thousands of users on free tier.
Agora Video SDK
Live fitness classes · 1-to-many
Same vendor as chat — single account, unified billing. Coach streams to 20+ members simultaneously. Handles poor UAE Wi-Fi gracefully. Replaces Microsoft Teams with a native in-app experience that doesn't lose the user to another app.
Firebase Cloud Messaging
Push notifications
Free, reliable, works on both platforms. Drives re-engagement: photos uploaded, class starting in 10 minutes, mandir live now, new group chat message. Push is the retention layer — without it, seniors forget to open the app.
Branch.io
Deep links · referral tracking
Every WhatsApp share contains a Branch link. Tracks: which member shared, which contact clicked, which signup came from which share. This data is how the loop stops being guesswork and becomes measurable.
Stripe
Subscription billing
AED 20/month recurring subscription via Stripe Billing. Supports UAE cards, Apple Pay, Google Pay, and local wallets. Stripe Checkout handles the entire payment UI — no PCI burden on the app. Webhooks sync subscription status to Supabase in real time.
YouTube IFrame API
Mandir arti live streams
Mandirs already stream on YouTube. Embedding their live URL is zero cost and zero hosting burden. The temple handles the stream — 2nd Innings just curates and surfaces. Flutter's webview plugin renders it natively.
Why this stack works for a solo dev
Every service here has: a Flutter SDK, good docs, a free tier, and runs without DevOps. No infrastructure to maintain, no servers to patch, no downtime responsibility. The dev's time goes into the app, not into keeping the backend alive.

The three-field rule
Name, phone (OTP verified), neighbourhood. Nothing else at signup. Every additional field drops 15–20% of users at this demographic. Language, faith, interests, health conditions, emergency contact, birthdate — all of these can be captured later through profile completion prompts spread across the first week of usage. Do not put them in the signup flow.
Contact permission — the hidden business engine
The contact list permission request is the single most valuable piece of user data in the entire product. Asked correctly, it unlocks:
  • Warm acquisition — app can show "3 of your contacts are already members" which is the highest-converting invitation mechanic known
  • Retention loop — when Ramesh's friend joins 3 weeks later, Ramesh gets notified — social proof keeps him active
  • Marketing data — phone numbers in contacts are potential subscribers. Carefully crafted WhatsApp outreach to "friends of existing members" converts at 10–15x cold outreach
  • Referral ranking — you can see which members have the most overlap with other members' contacts — those are your community super-nodes
The framing that unlocks permission
Do NOT ask: "Allow 2nd Innings to access your contacts?" — that triggers the OS system dialog with generic scary language.

DO show a custom pre-prompt first: "Find friends already here." The utility is clear, the benefit is social, and the "Skip" button is right there. Users who tap Allow in this custom UI then tap Allow in the OS dialog convert at 60–70%. Users who only see the OS dialog with no context convert at 15–20%.
Technical implementation
Auth
Supabase Auth with phone OTP. Uses signInWithOtp() Flutter method. Twilio under the hood — works with all UAE carriers. OTP delivery in 3–8 seconds.
Resend timer
Visible countdown at 20 seconds — not the default 60. Seniors will close the app thinking it's broken if there's no visible retry option quickly.
Contact API
Flutter flutter_contacts package. iOS requires NSContactsUsageDescription in Info.plist with the same friendly copy used in the custom prompt — Apple rejects apps where OS prompt and in-app prompt disagree.
Contact storage
Hashed phone numbers stored in Supabase table user_contacts. Never store raw names or emails — privacy requirement and App Store review pass. Match against users table to find overlaps.
OTP fallback
If SMS fails: WhatsApp OTP via Twilio WhatsApp API as secondary option. This demographic has WhatsApp on every phone — 99% delivery rate.
Growth angle — the compounding asset
The value of 100 members with contact permission granted is not 100 × average contact list. It is the overlap graph between them. If Ramesh has Priya in his contacts and Priya joins 2 months later from a different channel — the app can notify Ramesh automatically, which drives re-engagement. At 100 members, the overlap graph is already valuable. At 1,000 members in UAE, it becomes a defensible moat no competitor can replicate.

Business rationale — why chat is non-negotiable
Without chat, the app is opened once a week — on meetup day. With chat, the app is opened 3–5 times per day. That frequency shift is the difference between a forgotten subscription and a renewed one. For a AED 20/month product, daily engagement is what prevents churn. Retention is a bigger problem than acquisition at this price point.
Chat also replaces the WhatsApp groups that community organisers currently run informally. Today those groups exist on WhatsApp — meaning 2nd Innings has zero control, zero data, and zero product moat. Moving chat into the app: ops sees everything, can moderate, can surface content, can prompt members, and owns the conversation.
Chat structure
Neighbourhood chat
One chat per neighbourhood: Marina, JLT, Abu Dhabi, Sharjah. Auto-joined based on onboarding selection. This is the primary social space.
Activity chats
One chat per recurring class: Yoga Group, Coffee Club, Mandir Arti. Members join on first attendance. Low-volume, high-intent.
Meetup chats
Temporary chat for each meetup — auto-created 48hrs before, auto-archived 7 days after. Logistics, arrival coordination, post-event photos.
Ops broadcast
Read-only channel for official announcements. Next meetup, new class, mandir schedule. No member replies — avoids noise.

05
Moderation

Growth angle — the reactivation trigger
When an inactive member (no open in 7+ days) gets a push notification saying "Priya and Amita are talking in Marina Seniors" — that is the reactivation moment. Named people from their actual group, not generic app notifications. Personalised push based on chat activity has 8–10x the open rate of generic app notifications — because it feels like a real conversation they're missing, not an app begging for attention.

Feature 03 · Live Fitness Video
Replace Microsoft Teams with native in-app video
Fitness classes currently run on Microsoft Teams. This is a critical growth leak. Every class is an opportunity to reinforce the app as the home of the community — instead, members leave the app, open Teams, and Teams takes the credit in their mental model. Moving classes in-app via Agora Video SDK fixes this and unlocks features Teams can never offer.
Why Teams is hurting growth
Reason 1 — brand dilution: When a senior opens Teams three times a week for yoga, Teams becomes the habit. 2nd Innings becomes the middleman. If we're the middleman, we're replaceable.
Reason 2 — no engagement data: Teams doesn't tell us who joined, who stayed, who dropped off. Churn prediction is impossible without this data.
Reason 3 — no social moment: Teams has no chat that persists, no way to "say hi" to another attendee after class, no profile tap-through. All the relationship-building value of a live class is lost the moment the call ends.
Reason 4 — friction for new members: An invited friend who just subscribed now has to download Microsoft Teams separately, create a Microsoft account, and figure out the meeting link. Most will not.

Agora Video SDK — technical implementation
SDK
agora_rtc_engine Flutter package. Same Agora account as chat — unified billing. Integration is straightforward: join channel → render coach's video → subscribe to audio.
Mode
Live broadcasting mode (1-to-many), not video call mode. Coach is the host. Members are audience. Coach sees all members in small tiles. Members see coach full-screen.
Token auth
Server-side token generation in Supabase Edge Function. Only paying subscribers get a token — this is how access control works. Token expires after 2 hours.
Recording
Agora Cloud Recording starts automatically when coach joins. Saves to Supabase storage. Available in app as "Last week's class" within 1 hour of class end.
Bandwidth
UAE Wi-Fi is generally strong but 4G can be patchy. Agora auto-adapts video quality based on connection. Member on weak connection still gets audio even if video cuts.
Cost
Free tier: 10,000 minutes/month. At 30 members × 3 classes × 45 min = 4,050 minutes. Well within free tier for Month 1–2. Paid tier kicks in around 200 members.
Growth angle — first class free
For invited contacts who sign up but don't subscribe yet: give them one free class experience in-app. Not Teams — the native in-app experience. They feel the quality, see the other members, hear the coach call members by name. By the time the class ends, the AED 20 subscription is already decided. Free class attendance → subscription conversion is measurably 3–4x higher than just showing a subscription prompt.

Branch.io deep links — the measurement layer
01
Link generation
When member taps Share, app calls Branch SDK to generate a unique link. Link carries: member_id (who shared), event_id (which meetup photo), utm_source=whatsapp. Takes ~200ms.
02
Install attribution
Recipient taps link → Branch intercepts → if app installed, opens directly to the shared photo. If not installed, routes to App Store or Play Store. After install, Branch preserves the attribution through the entire signup flow — the new user's referral_member_id is known from the first open.
03
Dashboard analytics
Branch dashboard shows: which members' shares drive the most downloads, which WhatsApp contacts click links fastest, which photo themes (walks vs yoga vs coffee) convert best. This data guides every content decision downstream.
04
Cost
Branch free tier: 10,000 monthly clicks. At 100 members averaging 3 shares/month with 5 clicks each = 1,500 clicks. Well within free tier for entire Year 1.

Capture
Team member at meetup uses any phone. 15–20 candid photos per event — no posed group shots. Natural light only.

The share rate is highest in the first 2 hours after the push notification fires. The emotional moment is fresh, the member feels connected to the community, the photo captures a feeling. Delay the upload by a day and the share rate drops by 60%. This is why the 10-minute turnaround from event to app is non-negotiable — it is not a technical detail, it is the single most important operational discipline in the entire growth model.

YouTube IFrame API — technical implementation
01
Embedding streams
Supabase streams table stores YouTube video IDs and schedules. Flutter app uses youtube_player_flutter package to render the live stream. No API key needed for public YouTube embeds.
02
Live detection
Scheduled Supabase Edge Function runs every 15 minutes. Calls YouTube Data API v3 to check if scheduled streams are currently live. Updates streams.is_live flag. App only shows the Live badge when confirmed live — no false notifications.
03
Push notifications via Firebase
When is_live flips to true, Edge Function triggers Firebase Cloud Messaging. Push targets users by preference: Hindu members get mandir pushes, Muslim members get mosque pushes. Faith preference set via profile prompt after first week.
04
In-stream chat via Agora
Each stream has a temporary Agora chat group. Members watching can send 🙏 emoji, "Jai Shri Krishna", "Amen", etc. Creates a shared spiritual moment with other members — the feeling of praying together, even from different homes.
05
Scheduled reminders
Members tap "Remind me" → Firebase FCM schedules a notification for 5 minutes before the stream. Works even if app is closed. The reminder is what drives the daily return visit.
Why not record and replay?
Arti is a live spiritual act, not content to binge. The liveness is the value. Watching a recorded arti at midnight is not the same experience. Keep streams live-only. Add a "Daily spiritual calendar" screen that shows what's streaming when — but resist the urge to build a video library. This feature is about being present, not consuming archives.
Growth angle — the re-activation moment
When a member hasn't opened the app in 3 days — send a push: "Morning arti starting in 5 minutes." This is the highest-converting reactivation message possible for this demographic. It combines habit, spirituality, and community. Measured across similar apps: 45–55% of lapsed users who receive a well-timed arti push open the app and stay for 8+ minutes. Zero other re-engagement tactic comes close.

The compounding insight
By Day 30, the system has 4 retention mechanisms running in parallel: daily mandir push (14x/week), group chat activity (5x/day), weekly fitness classes (3x/week), weekly meetup photos (1x/week). No single feature carries the weight. Miss one and the user still has three reasons to open the app. This is why the feature set is what it is — not because more is better, but because resilience comes from multiple weak signals, not one strong one.

Technical implementation
Stack
Next.js + Supabase JS client. Single-page web app deployed to Vercel free tier. Same Flutter dev builds this — same Supabase backend, shared auth, reuses the data layer.
Auth
Supabase Auth with email magic link. Only whitelisted ops emails can sign in. Row-level security policies ensure admin can only read/write admin-scoped data.
Deployment
Vercel. Free tier sufficient at this scale. Custom domain: admin.2ndinnings.app. Auto-deploys from GitHub on every commit.
Build time
5–6 days of dev work after main app core is complete. Built during Week 3 in parallel with app polish. Ships by Day 21.
Future extensions
Post-100 users: add A/B testing dashboard, coach self-service scheduling, automated churn playbooks, revenue cohort analysis. None of these are needed for the first 100.
The scope discipline
Every admin feature beyond these 10 modules is a post-100 problem. Resist the urge to build analytics, cohort tools, or workflow automations now. The goal is: ops team runs the business without calling the dev. Not: ops team has every possible analytics view. Ship the minimum, measure what they actually use, expand based on evidence.
